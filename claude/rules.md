# 语言要求

- 使用英文思考，回复我时使用中文。

# 沟通方式

- 说人话，少用术语；必须用术语时，用一句话解释清楚。
- 先给结论，再给关键依据和下一步。
- 解释或实现复杂流程时，说明原理和执行步骤。
- 需要流程图、时序图、架构图或 Mermaid 图时，优先使用 `mermaid-kitty` skill 输出 ASCII 渲染结果，不直接贴原始 Mermaid 代码。

# 修改代码流程

- 动手前先阅读相关代码、调用链和配置，覆盖本次改动会影响的范围即可。
- 改动保持最小，优先复用现有结构、风格和工具。
- 代码要简单清楚：函数尽量小，避免重复，控制圈复杂度。
- 只有在能明显降低复杂度时，才引入抽象、设计模式或跨模块调整。
- 修改后检查是否有报错；如果项目能编译或测试，就运行对应的编译/测试命令。
- 涉及逻辑改动时，按风险验证主路径、关键边界和错误/不命中情况；无法验证时说明原因和剩余风险。

# 工作区安全

- 修改只限本次相关文件；发现无关已有改动，不回滚、不覆盖。
- 禁止使用 `git reset --hard`、`git checkout --` 等破坏性命令，除非我明确要求。
- 回滚或清理只处理本轮自己产生的改动；无法区分时先说明情况。

# 函数拆分边界

- 小改动优先保持就地表达；不要为了“函数尽量小”把 1-2 行简单条件拆成多层 helper。
- 只有在明显提升可读性、存在复用、需要单独测试、或能显著降低复杂度时才抽函数。
- 一次性业务特判最多抽一个语义清楚的函数，避免 helper 套 helper。
- 不做预判式抽象；只有出现第二个真实用例时，再考虑通用化。

# 单元测试规则

- 涉及逻辑改动时，优先补单元测试；至少覆盖正常情况、边界情况、错误/不命中情况中的关键路径。
- 以下情况必须补单元测试，除非项目没有测试框架或测试成本明显不合理：
  - 修复 bug，尤其是历史线上问题或容易回归的问题。
  - 新增或修改条件判断、路由、过滤、频控、权限、状态流转、数据映射等逻辑。
  - 修改公共函数、公共组件、协议转换、配置解析、序列化/反序列化。
  - 涉及金额、账号、设备、推送、人群、厂商通道等容易误伤用户的逻辑。
  - 用户明确要求补测试。
- 以下情况可以不新增单元测试，但要运行已有编译/测试或做轻量验证：
  - 纯文案、注释、日志格式调整。
  - 只改静态配置，且配置有现网发布校验或人工审核。
  - 简单样式调整或无逻辑的 UI 微调。
  - 临时排查用改动，结束前会恢复，且不会提交或发布。
  - 测试框架缺失、依赖外部环境严重、补测试成本明显高于改动风险。
- 不为了覆盖率堆测试；单测要验证真实行为和关键边界，不测私有实现细节。
- 如果没有补单测，回复里说明原因、已做的验证方式，以及剩余风险。

# Bug 修复流程

修复 Bug 时先判断风险：

- 小问题可以轻量处理：理解问题、定位原因、最小修复、验证结果。
- 复杂或高风险问题需要完整流程：复述问题、列出至少两种可能原因、说明验证方式和修复方案，并在修改前请求确认。

修复完成后：

- 自查改动范围，说明改了什么、为什么这样改。
- 做回归测试；如果验证不通过，先回滚本轮自己产生的改动再重新分析。

# 设计原则

- 简单优先：优先选择简单、可读、可维护的方案；不添加当前需求之外的推测性功能。
- 低耦合、单一职责：一个模块或函数尽量只做一件清楚的事；SOLID 按需使用，目标是降低耦合和复杂度，不是为了套原则。

# 不确定时的行为

- 不确定但可通过阅读代码、运行测试、查官方文档解决时，先自行验证。
- 只有涉及破坏性操作、多个方案代价接近、或业务风险明显时，才先说明已知信息和可选方案，确认后再动手。

# 文档协议

- 不写不必要的文档、修改记录或使用说明，除非明确要求。
- 文档必须和代码同步，不保留过时引用。
- Markdown 标题层级和章节格式保持一致。
- 文档里的代码片段必须能执行，且反映真实用例。
- 面向 API、工具或配置的文档，应说明目的、用途、参数和示例。
